# 电池接管升级规划 v0.3.0

## 1. 调研结论

### 1.1 真实项目边界

用户可见目录 `/Users/mahaoxuan/Desktop/产品经理/电池接管` 当前主要是运行资产目录，包含历史日志、历史 SQLite 数据库和空报告目录。真实可运行应用位于：

```text
/Users/mahaoxuan/Library/Application Support/BatteryTakeover/app
```

桌面启动器 `电池接管-launcher.applescript` 会进入该目录，启动 `./btake --config ./config/default.toml dashboard --host 127.0.0.1 --port 8775`，再打开本地 Dashboard。

关键风险是数据源分裂：桌面目录和 App Support 目录各有 `battery.db`。当前启动器和真实服务使用 App Support 目录下的配置与数据库，桌面目录更像历史运行资产或旧状态副本。

### 1.2 产品方向

基础“设定充电上限”已经不再是强差异点。Apple 在 macOS Tahoe 26.4+ / Apple silicon 上提供系统级 Charge Limit，范围为 80%-100%；`batt` 也继续提供更细的 charge limit、上下限和睡眠相关处理。

因此本项目应升级为：

```text
可信、可解释、可恢复的本地电池接管台
```

它的差异点不是替代所有电池工具，而是把当前为什么充电、为什么停止、何时恢复、是否降级、哪里出错讲清楚，并留下可审计记录。

### 1.3 交付风险

项目已经有入口、测试、构建和真实运行能力。当前机器上 `batt` 可用，`battery` 不存在，项目正在真实影响本机充电策略。

交付风险集中在：

- 真实写入后端的安全边界。
- LaunchAgent 是否重复启动或残留。
- Dashboard 本地 API 无鉴权，只能绑定 `127.0.0.1`。
- dry-run 必须不触发 `batt limit` 或 `batt disable`。
- 关闭项目管理必须清除项目限充并交还系统/后端。
- 报告和 Dashboard 不能误导电池健康或承诺硬件旁路能力。

## 2. v0.3.0 目标

v0.3.0 不重写底层策略，不引入新 GUI 框架，先把现有 Web Dashboard 升级成可用控制台。

验收标准：

1. Dashboard 第一屏能回答“现在为什么这样”。
2. Dashboard 明确展示目标区间、Agent 状态、权威 DB/log/report 路径。
3. 24h 曲线显示停充线与恢复线。
4. 24h 摘要展示样本、动作、失败、区间内占比。
5. 日报摘要优先，不再默认输出上千行动作流水。
6. 新增逻辑有单元测试覆盖。
7. 本地 smoke 能验证 doctor/status/dry-run/dashboard API。

## 3. 交付范围

### P0 本轮完成

- 新增 `/api/snapshot`：合并 overview、policy、24h 摘要和当前状态解释。
- Dashboard 改为状态控制台：
  - 当前电量
  - 控制状态
  - 目标区间
  - Agent 状态
  - 今日判断
  - 阈值控制
  - 24h 摘要
  - 权威运行路径
  - 最近动作
- 日报改为摘要优先：
  - 今日结论
  - 区间内样本占比
  - 动作成功率
  - 失败明细
  - 最近动作限量展示
- 保留现有 `batt`/`battery` 执行器行为，不扩大真实写入面。

### P1 下一轮

- CLI smoke 测试脚本，使用临时配置和临时 state/log/reports。
- fake `batt` 集成测试，验证 limit/disable/dry-run/降级。
- Dashboard HTTP 层测试：非法阈值、非法 JSON、端口占用、localhost 绑定。
- 启动器增强：启动前检查 dashboard/agent 状态，并提示当前使用的权威数据路径。
- 桌面目录 manifest，标明历史 DB 与主运行 DB 的关系。

### P2 后续产品化

- 原生菜单栏 Popover。
- 出门充满一次，并自动恢复保护。
- macOS 原生 Charge Limit / Optimized Charging 冲突检测。
- 安装包验收与卸载恢复测试。
- 诊断报告导出。

## 4. 安全边界

- Dashboard 默认只能绑定 `127.0.0.1`。
- `/api/config` 和 `/api/enforce-now` 会触发真实动作，不能开放到局域网。
- `enforce --dry-run` 必须只评估策略，不写后端。
- 失败后必须能降级只读，不反复写。
- 关闭项目管理必须明确恢复系统或底层后端控制。
- UI 文案只描述系统读数和策略行为，不宣称修复电池、硬件旁路或保证寿命收益。
